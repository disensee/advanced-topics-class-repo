Fundamental Differences Between JavaScript and Java

If you've had a little bit of experience with both Java and JavaScript, then this article may help to clear up some of the differences the two languages. A word of caution: one thing about programming that can be very confusing for beginners is that some of the terms are used loosely. And a term may mean something different depending on which language you are speaking about. This 'term confusion' is apparent in many of the topics discussed in in this article. I would actually bet that even many experienced programmers are confused about some of the terms mentioned below, such as: static, strongly typed, dynamic, loosely typed.

Java is Compiled (and Interpreted), JavaScript is Interpreted

In Java, source code is compiled into bytecode. Then, any computer that has the Java Runtine Environment (aka the JVM) can run/execute the bytecode. The JVM interprets the byte code into machine code.

JavaScript code is not compiled. Instead the source code is typically sent from a web server to a browser. The browser then interprets the source code into machine code, and then executes the machine code. Remember that all code (whether it's source code, byte code, or some other type of code) must eventually be translated into machine code before a computer can execute it.

What is the difference between 'compiled' and 'interpreted'? A compiler will check your code for correctness before converting it from source code to some type of binary code, it does not execute the program. An interpreter is a program (such as a browser, or the JVM) that will convert some type of code into machine code and then execute the machine code. An interpreter is also known as a 'runtime environment'. So the JVM is the runtime environment for Java byte code, and a browser is the run time envirnment for code used in web pages (HTML, CSS, and JavaScript).

Loosely Typed vs Strong Typing

In JavaScript you don't have to specify the data types for variables (variables are loosely typed).

// JavaScript

var x = 7;

In Java you must specify the type of data that a variable will store when it is declared (variables are strongly typed).

// Java

int x = 7;

In JavaScript you can assign any type of data to a variable (some call this dynamic typing):

// JavaScript

var x = 7;      // x is storing a number
x = "hello";    // now x is storing a string

In Java, the compiler will not allow you to change the data type that a variable is storing (some call this static typing). 

// Java

int x = 7;
x = "hello";  // Will not compile! You are trying to shove a string into x, which was declared to store ints.

In JavaScript you can pass any data type as an argument.

// JavaScript

function add(x,y){
   alert(x + y);
}

add(2,3);
add("hello", "world");
add(1, "hello");

In Java you must pass in the proper data type.

// Java

public void add(int x, int y){
   System.out.println(x + y);
}

add("hello", "world"); // Will not compile! (the add() method requires both params to be ints)

Note how using strong typing (and compiling) can prevent runtime errors. In Java, if you misuse a method by passing in the wrong type you'll get a compile error. But in JavaScript, you can actually run the program, if the parameters are not the right data type, then the function may not work as intended.

Dynamic Objects vs Predefined Classes

In JavaScript, objects can be created and defined at run time (objects are dynamic).

// JavaScript

var obj = {}; 				// just created an object (no class needed!)
var obj.firstName = "Bob"; 	// just added a property to the object

var propName = prompt("What property would you like to add to the object?");
var propValue = prompt("What value would you like to assign to the property");

obj[propName] = propValue;

Java objects must be created from predefined classes. The compiler ensures that every object is created from a class (some people call this 'static typing' - I'm actually still a bit confused about what static typing really means!).

JavaScript is Functional (Higher Order Functions)

JavaScript is considered a 'functional' language, which means that you can do things with functions that other languages don't allow. Functions in JavaScript are known as 'higher order'. Some people say that in JavaScript, functions are treated as 'first class citizens' because you can do some of the same things with them that you can do with variables. Note that over the years, there has been a developing trend toward functional programming. As Java has evolved, it has begun to incorporate some of the features of functional programming.

There are 3 basic features of higher order functions:

  1. Functions can be assigned to variables.
  2. Functions can be used as paramters.
  3. Functions can be returned.

Functions can be assigned to variables (and to properties of objects).

// JavaScript

var fn = function(){
   alert("foo!");
};

// to invoke the function...
fn();

Functions can be passed in as params.

If you've gone beyond just scratching the surface of JavaScript, then this code will look familiar to you:

window.addEventListener("load", function(){ /*code goes here*/ });

This code sample invokes the addEventListener() method of the window object. Notice that addEventListener() takes to parameters, this first is a string, and the second is an function. The function will be invoked by the browser when the window triggers the 'load' event.

Functions can be used as return values.

var passFn = function(){
   alert("You passed!");
};

var failFn = function(){
   alert("You failed!");
};

function getResult(grade){
   if(grade > 70){
      return passFn;
   }else{
      return failFn;
   }
}

var resultFn = getResult(75);
resultFn();

Functions can be nested inside of other functions

In Java, you cannot declare/define a method inside another method. But in JavaScript it's very common to 'nest' functions. 

window.addEventListener("click", function(){
   
   function sayHello(){
      alert("Hello!");
   }

   sayHello();

});

An Important Similarity between Java and JavaScript

JavaScript and Java behave the same when it comes to variables that store primitives and variables that refer to objects. 

var x = 7;		// The variable x is storing the value 7
var obj = {};		// The variable obj is storing a memory address of an object.

Note that object variables are said to be 'references' because that store a memory address that refers to an object.

Remember from Intro to Java that when you pass a primitive variable as a parameter to a method, a copy of the variable's value is created so as to leave the original value unchanged. Here is a JavaScript code sample that demonstrates the same behavior. 

var x = 7;

function someFn(param){
   param += 3;
}

someFn(x);	// A copy of the value of x is created 
alert(x);  	// x is still 7

But parameters that refer to objects are different, what's really getting passed is a copy of a memory address (the address of the object). So, in the code sample below both obj and param are refering to the same memory address. 

var obj = {name: "Bob"};

function someFn(param){
   param.name = "Robert";
}

someFn(obj);		// A copy of the memory address pointing to the object is created
alert(obj.name); 	// Shows 'Robert'

I'll finish this article off with a bit more 'term confusion'. Most languages behave the same as both Java and JavaScript when dealing with parameters that store primitives and parameters that store objects. This is commonly known as 'primitives are passed by by value and objects are passed by reference'. But if you dig around the web you'll find that, in Java, parameters are always passed by value! How confusing is that!? But remember that the value of a variable that refers to an object is a memory address. So when you pass an object into a method, a copy of the memory address is created. Both copies of the memory address refer to the same object.
